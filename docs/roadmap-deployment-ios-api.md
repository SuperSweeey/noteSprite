# NoteSprite 部署、iOS 与开放收藏 API 方案暂定版

## 1. 文档目的

这份文档用于把 NoteSprite 下一阶段的三个方向先收成统一底稿：

1. 把当前本地运行版本部署到云服务器，支持我自己远程访问。
2. 准备 iOS 端形态，让手机上也能进入同一套笔记与 AI 工作流。
3. 设计一个外部收藏 API，让手机、电脑和第三方工具都能快速把内容送进 NoteSprite。

这不是最终技术定案，而是当前版本基础上的执行暂定版。后续开发、拆任务、排优先级、做验收，都以这份文档为背景依据。

## 2. 背景

当前 NoteSprite 已经从本地原型走到可持续开发阶段，已经具备这些基础能力：

1. Web 端主流程可用：首页收集、笔记详情、知识库、AI 对话、设置页。
2. 外部输入已经统一到“丢万物”入口，支持文本、链接、图片、音频、视频进入系统。
3. 音视频可接入本地上传与转录链路，图片笔记可以保留原图并形成笔记。
4. AI 侧边栏、AI 页面、AI 解读、知识库上下文注入等核心链路已经有回归测试和稳定性修复。

现在的主要限制也很清楚：

1. 运行环境还是偏本地开发机，不适合长期远程使用。
2. 当前交互主要面向桌面浏览器，移动端和 iOS 还没有正式产品形态。
3. 外部输入虽然已有页面入口，但还没有一个稳定、可编程、可跨设备调用的 API 层。

## 3. 现有情况

### 3.1 当前技术栈

1. 前端与服务端：Next.js App Router。
2. 数据层：Prisma。
3. 当前数据库：SQLite。
4. AI 与业务配置：以应用内设置和服务端配置解析为主。
5. 转录链路：Node 侧调起仓库内 `python/main.py`，依赖 Python、ffmpeg、OSS、转录服务配置。
6. 当前上传与外部输入：`/api/drop`、`/api/transcribe`、`/api/transcribe/upload`。

### 3.2 当前适合保留的部分

1. 单一 Web 代码库是优势，后续云部署和 iOS 封装都能复用。
2. “丢万物”已经形成统一输入语义，适合作为 API 化的基础。
3. AI 对话和笔记体系已经在同一个 Prisma 数据模型里，天然适合多端同步。

### 3.3 当前不适合直接沿用到下一阶段的部分

1. SQLite 更适合本地开发或轻量单机，不适合作为长期多端同步主库。
2. 上传文件、图片、音视频、转录中间产物还缺少正式的持久化与备份设计。
3. 现在的输入入口偏页面行为，还不是独立的通用 ingest 能力。
4. 现有 Web 端虽然能在浏览器访问，但还不是移动优先体验。

## 4. 总目标

下一阶段不是同时做三套完全独立的新系统，而是围绕一条主线推进：

先把 NoteSprite 变成“一个可长期运行的个人云笔记服务”，然后让 Web、iOS、快捷指令、桌面工具都接入这同一套后端能力。

也就是说：

1. 云服务器版是底座。
2. iOS 版是这个底座的移动入口。
3. 收藏 API 是这个底座的外部输入总线。

## 5. 方案一：个人云部署

### 5.1 目的

把当前只能在本地使用的 NoteSprite 变成一个只对我自己开放、但能从任何设备访问的私有服务。

### 5.2 要求

1. 只给自己用，不做公开注册，不做多人租户。
2. 远程访问稳定，不能依赖本地电脑一直开着。
3. 上传文件、图片、音视频、转录结果都能持久保存。
4. 具备最基本的备份和恢复能力。
5. 访问必须有明确鉴权，不能直接裸露公网。

### 5.3 暂定方案

第一阶段采用“单用户私有部署”：

1. 服务器环境：Ubuntu LTS、Node.js、Python、ffmpeg、pm2 或 systemd。
2. 应用部署：用当前 Next.js 服务端直接部署，不拆前后端。
3. 数据库：建议直接切 PostgreSQL，避免后面 iOS、多端写入、外部 API 并发时再返工。
4. 文件存储：笔记附件、上传源文件、图片、音视频、转录产物使用服务器持久目录；后续可再迁 OSS。
5. 访问控制：优先考虑 Tailscale 私网访问；备选是 Cloudflare Tunnel + Access。
6. 反向代理与 HTTPS：用 Nginx 或 Caddy 承接域名、HTTPS 和转发。
7. 备份：PostgreSQL 定时备份，附件目录定时打包或同步到对象存储。

### 5.4 执行顺序暂定

1. 先做部署环境清单和 `.env` 变量收口。
2. 把 SQLite 迁到 PostgreSQL。
3. 梳理上传目录、附件目录、Python 输出目录的持久化路径。
4. 写部署脚本或部署文档。
5. 上服务器试跑。
6. 接入私有访问和 HTTPS。
7. 最后补备份与恢复演练。

### 5.5 这条方案的判断

这条方案最符合“只自己用，但要长期可访问”的目标。复杂度远低于公开 SaaS，但已经足够支撑后面的 iOS 与外部 API。

## 6. 方案二：iOS 端

### 6.1 目的

让手机上也能进入同一套 NoteSprite 工作流，包括查看笔记、和 AI 交互、快速收藏内容。

### 6.2 要求

1. 与云端同一份数据同步，不做本地孤岛版本。
2. 第一版重点是“能用”，不是先追求完全原生重写。
3. 必须支持手机上的输入行为，尤其是分享、上传、拍照、图片和链接收藏。
4. 不能为了做 iOS 端而复制一套业务逻辑。

### 6.3 暂定方案

采用“两层推进”：

第一层：先把当前 Web 端做到移动可用。

1. 首页输入和“丢万物”入口移动端可用。
2. 笔记详情页、AI 对话、知识库页面在手机宽度下可用。
3. 上传、预览、重试、错误提示在移动端语义清楚。

第二层：再做 iOS App 壳。

1. 优先选 Capacitor，把当前 Web 应用打包进 iOS。
2. 原生能力重点接这几项：文件选择、系统分享进来、图片/音视频上传、登录态持久化。
3. 如果后续对体验要求更高，再评估 SwiftUI 原生重写。

### 6.4 为什么先不直接原生重写

1. 当前核心价值在业务链路，不在原生 UI 特效。
2. 现在后端和 API 还在演进，先重写 iOS 会放大维护成本。
3. Capacitor 方案能最快验证“手机上到底怎么用最频繁”。

### 6.5 第一版 iOS 的实际目标

1. 登录自己的 NoteSprite 服务。
2. 查看和搜索笔记。
3. 用“丢万物”收藏链接、图片、音视频。
4. 进入笔记详情和 AI 继续对话。
5. 从系统分享菜单直接送内容进 NoteSprite。

## 7. 方案三：外部收藏 API

### 7.1 目的

把当前“只能通过页面输入”的外部收集，升级成“任何设备、任何工具都能调用”的标准入口。

### 7.2 要求

1. 链接、文本、图片、音频、视频都可以统一进入。
2. 鉴权必须独立，不能依赖浏览器 Cookie。
3. 能被 iPhone 快捷指令、分享扩展、桌面工具、浏览器插件、自动化工具调用。
4. 异步任务必须可追踪，不能只回一句成功。

### 7.3 暂定接口方向

建议新增一层统一 ingest API，例如：

`POST /api/ingest`

支持的输入：

1. `text`
2. `url`
3. `image`
4. `audio`
5. `video`
6. `source`
7. `title`
8. `tags`
9. `knowledgeBaseId`

建议支持两种提交模式：

1. `multipart/form-data`：适合图片、音频、视频和混合上传。
2. `application/json`：适合文本、链接、自动化工具、快捷指令。

### 7.4 鉴权方案暂定

不建议直接复用网页登录态，建议单独做 Personal Access Token：

1. 用户在设置页生成或吊销 token。
2. 外部工具通过 `Authorization: Bearer <token>` 调用。
3. token 支持备注用途，例如“iPhone 快捷指令”“Raycast”“浏览器扩展”。

### 7.5 后端实现方向

后端不要再把页面逻辑和 API 逻辑绑死，应该抽成统一 ingest service：

1. 页面“丢万物”调用这个 service。
2. 外部 API 也调用同一个 service。
3. service 内部统一判断输入类型，决定是直接建笔记、图片理解、还是转录任务。

### 7.6 返回结果设计暂定

1. 立即完成型：直接返回 `noteId`、`status=completed`。
2. 异步处理型：返回 `jobId`、`status=processing`、查询地址。
3. 失败型：返回标准错误码和人话错误信息。

### 7.7 第一批接入方

1. iPhone 快捷指令。
2. iOS 分享扩展。
3. Mac 上的 Raycast 或 Alfred。
4. Windows 上的快捷工具或脚本。
5. 浏览器扩展。
6. 任意支持 webhook 的第三方工具。

## 8. 三者之间的关系

这三个方向不能各自独立开发，应该按依赖关系推进：

1. 云部署先完成，形成稳定后端。
2. 外部收藏 API 作为统一输入总线先抽出来。
3. iOS 端再接入这套统一 API。

否则会出现两个问题：

1. iOS 和 Web 各写一套输入逻辑，后续维护会非常重。
2. 本地版、云版、移动版三套状态不一致，越做越散。

## 9. 当前建议优先级

### P1：先做

1. 云部署方案落地文档与环境脚本。
2. PostgreSQL 迁移。
3. 附件与上传目录持久化方案。
4. 统一 ingest service 抽象。
5. Personal Access Token 设计。

### P2：第二批做

1. 正式的 `POST /api/ingest`。
2. iPhone 快捷指令调用链路。
3. Web 移动端适配。
4. iOS 分享入口原型。

### P3：第三批做

1. Capacitor iOS 壳。
2. 移动端上传与分享体验打磨。
3. 后续原生化评估。

## 10. 暂定结论

当前最合理的路线不是“先做 iOS”，而是：

1. 先把 NoteSprite 变成一个稳定的个人云服务。
2. 再把“丢万物”真正抽成 API 能力。
3. 然后让 iOS、快捷指令、桌面工具都来调用这套统一能力。

这样做的好处是：

1. 后端能力只建设一次。
2. 数据与同步模型从一开始就是统一的。
3. 你以后想接更多入口，不需要重做底层。
