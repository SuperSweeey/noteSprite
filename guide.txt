下面是根据你补充后的新版完整产品报告。我把几个关键点已经修正进去了：

第一，**“信息”不再只理解为链接或平台内容，而是扩展为外部输入与文件资产**，包括视频文件、音频文件、PDF、图片、截图、网页、文章、链接等。

第二，**标签系统明确完全仿照 flomo 的 `#a/b/c` 分级标签模式**，不做复杂的传统标签管理器，而是让标签自然地出现在笔记正文中，并自动解析。

第三，**多端同步明确为手机端 + 网页端优先**，建议采用“服务端同步 + 本地缓存 + Markdown/元数据双层存储”的方案。早期不建议一上来做过于复杂的纯本地优先 CRDT 同步，先保证可用、稳定、可自托管。

### **开源 AI 笔记系统产品报告**

报告日期：2026 年 5 月 9 日  
项目暂定名：**OpenNote AI**  
产品方向：**开源、自托管、AI 原生、多端同步的个人知识管理系统**

---

## **一、产品背景**

用户正在越来越频繁地使用 AI 笔记、知识库、剪藏工具和个人知识管理软件，但现有工具普遍存在几个问题。

一类产品体验很好，但数据锁定严重。用户的笔记、想法、资料和 AI 分析结果长期存储在商业平台里，一旦产品停止运营、涨价、限制导出，用户的个人知识资产就会面临风险。

另一类开源笔记工具足够自由，但 AI 能力、移动端体验、多源采集能力和复习机制较弱。它们更像文件管理器或 Markdown 编辑器，而不是一个真正帮助用户理解、整理和记住知识的系统。

还有一类 AI 知识库产品能处理文档和问答，但不适合作为日常笔记工具。它们通常更偏“资料问答”，缺少快速记录、标签、复习、灵感沉淀和长期个人知识管理能力。

因此，本项目希望做一个开源版的 AI 笔记系统，让用户既能获得类似现代 AI 笔记产品的便利，又能保留对数据、部署方式和知识资产的控制权。

---

## **二、产品定位**

OpenNote AI 是一款开源 AI 笔记系统，面向重度信息输入和个人知识沉淀用户，支持快速记录想法、多源信息收集、文件资产管理、Markdown 统一沉淀、flomo 式分级标签、知识库体系、AI 总结分析、AI 自动整理、多端同步和每日复习。

它不是一个单纯的 Markdown 编辑器，也不是一个简单的收藏夹，而是一个完整的个人知识工作流系统。

它要解决的核心问题是：

> 帮助用户把每天产生的想法、看到的信息、保存的文件和长期学习资料，统一沉淀为可搜索、可整理、可复习、可问答、可导出的个人知识资产。

产品关键词是：

**开源、自托管、Markdown、AI、收集箱、分级标签、知识库、多端同步、复习。**

---

## **三、产品愿景**

OpenNote AI 的长期愿景是成为个人知识管理的开源基础设施。

用户不应该因为某个商业笔记产品倒闭、涨价、封闭或导出困难而失去自己的长期知识积累。笔记系统应该像个人硬盘、个人博客、个人代码仓库一样，是用户真正拥有的数字资产系统。

这个产品的最终形态不是“帮用户多存一些东西”，而是帮助用户完成一个完整闭环：

> 捕捉想法 → 收集信息 → 处理文件 → 转化 Markdown → AI 理解 → 标签归类 → 知识库沉淀 → 每日复习 → 长期内化 → 再次输出。

---

## **四、目标用户**

第一阶段的目标用户不是所有人，而是高频记录、高频学习、高频收集信息的人。

典型用户包括产品经理、程序员、研究者、学生、内容创作者、自媒体从业者、创业者、知识管理爱好者、AI 工具重度用户，以及正在使用 flomo、Notion、Obsidian、Get 笔记、Readwise、Heptabase 等工具的人。

这些用户有几个共同特征。

他们每天会产生大量碎片想法，希望有一个低摩擦的记录入口。他们会阅读网页、文章、PDF、社交媒体内容，也会保存视频、音频、截图、图片等资料。他们不满足于简单收藏，而是希望这些内容可以被 AI 总结、被重新组织、被搜索、被复习，并最终成为自己的长期知识体系。

同时，他们对数据拥有权比较敏感。他们希望笔记能导出为 Markdown，希望可以自托管，希望即使项目未来停止维护，自己的数据仍然可以迁移到其他系统中。

---

## **五、核心用户需求**

### **1. 快速记录个人想法**

用户希望像使用 flomo 一样，随时打开应用就能记录想法，不需要先创建页面、选择模板、选择文件夹或填写复杂属性。

这类内容通常是用户自己的输出，例如灵感、感受、复盘、观点、选题、读后感、待办想法、产品想法、学习理解等。

这个场景的关键不是编辑能力多强，而是记录摩擦足够低。

产品应该支持打开即写、保存即走、自动识别标签、自动生成标题、自动摘要和后续整理建议。

---

### **2. 收集外部信息与文件资产**

你补充的这一点非常重要。

这里的“信息”不能只理解成网页链接或平台内容，而应该扩展为一切来自外部的输入，包括网页、文章、PDF、图片、截图、音频文件、视频文件、文档文件、语音记录、聊天记录、电子书摘录、课程资料等。

这些内容本质上是用户的**文件资产**和**信息资产**。

系统应该允许用户把这些资产统一放进 OpenNote AI，并尽可能转化为 Markdown 或结构化文本。

不同类型的资产处理方式不同。

| 类型 | 示例 | 处理方式 | 结果 |
|---|---|---|---|
| 网页文章 | 博客、新闻、教程 | 正文提取、清洗、转 Markdown | Markdown 笔记 |
| PDF | 论文、报告、电子书片段 | 文本提取、分页引用、AI 摘要 | Markdown 笔记 + 原文件 |
| 图片 | 截图、照片、海报 | OCR 识别、描述生成 | Markdown 笔记 + 图片 |
| 音频文件 | 录音、播客、访谈 | 语音转文字、分段总结 | 转录稿 + 摘要 + 原音频 |
| 视频文件 | 课程、会议、讲座 | 音轨提取、转写、关键帧/章节摘要 | 转录稿 + 摘要 + 原视频 |
| 链接 | 视频链接、文章链接 | 抓取元信息、提取正文或字幕 | Markdown 笔记 |
| 文档 | DOCX、TXT、MD | 文本解析、格式转换 | Markdown 笔记 |
| 手动输入 | 灵感、想法 | 直接保存 | Markdown 笔记 |

因此，产品里应该有一个明确的概念：

> OpenNote AI 不只是笔记系统，也是个人信息资产和文件资产的入口。

但为了保证第一版可实现，文件资产能力要分阶段做。MVP 可以先支持文字输入、Markdown、网页、PDF、图片，音频和视频文件放到 P1 或 P2 阶段。

---

### **3. 使用 flomo 式分级标签**

你明确说标签系统要完全仿照 flomo 的 `#a/b/c` 模式。这一点我建议作为产品的核心设计原则之一。

也就是说，用户不是在一个复杂的弹窗里选择标签，而是可以直接在正文里写：

`#产品/AI笔记/竞品分析`

`#学习/编程/React`

`#写作/选题/知识管理`

`#生活/复盘/每日记录`

系统自动解析这些标签，并把它们变成可点击、可筛选、可聚合的标签路径。

这种标签系统有几个优点。

第一，它足够自然。用户写笔记时顺手输入标签，不需要中断思路。

第二，它天然支持分级。`#产品/AI笔记/竞品分析` 同时属于 `#产品`、`#产品/AI笔记` 和 `#产品/AI笔记/竞品分析`。

第三，它适合碎片化记录。用户不需要像文件夹一样提前设计结构，而是在长期记录中自然生长出自己的标签体系。

第四，它方便 AI 参与整理。AI 可以根据笔记内容建议一个或多个分级标签，用户只需确认。

标签系统建议遵循以下规则：

| 规则 | 说明 |
|---|---|
| 标签以 `#` 开头 | 例如 `#产品` |
| 使用 `/` 表示层级 | 例如 `#产品/AI笔记/需求分析` |
| 标签可出现在正文任意位置 | 系统自动解析 |
| 一条笔记可有多个标签 | 例如同时包含 `#产品/AI笔记` 和 `#创业/工具` |
| 点击父级标签可查看子级内容 | 点击 `#产品` 能看到所有产品相关笔记 |
| 标签不强制提前创建 | 用户写了就自动生成 |
| 支持 AI 推荐标签 | 但最终由用户确认 |
| 支持标签重命名和合并 | 避免长期使用后标签混乱 |

产品里不建议设计成传统的“标签管理后台优先”。标签应该首先存在于内容中，然后系统再提供标签浏览、标签搜索、标签合并和标签统计能力。

---

### **4. 知识库与文件夹体系**

标签解决的是横向索引问题，但你仍然需要知识库体系来承载长期主题。

比如：

- AI 产品研究
- 个人成长
- 编程学习
- 商业案例
- 写作素材
- 创业项目
- 读书笔记
- 课程学习
- 工作资料

知识库更像一个大的主题空间。每个知识库里可以有文件夹、笔记、文件资产和 AI 问答。

标签和知识库的关系应该是并行的，而不是互相替代。

知识库适合长期归档，标签适合灵活关联。

例如一条笔记可以属于“AI 产品研究”知识库，同时拥有：

`#产品/AI笔记/功能设计`

`#创业/开源项目`

`#知识管理/复习系统`

这条笔记既有稳定归属，也能被多个主题索引。

---

### **5. AI 总结、分析与整理**

AI 是产品的核心效率层。

每条笔记或文件资产进入系统后，都应该能够被 AI 处理。

对于文字笔记，AI 可以生成标题、摘要、要点、关键词、标签建议和知识库建议。

对于网页文章，AI 可以提取核心观点、生成结构化大纲，并标记值得保存的内容。

对于 PDF，AI 可以生成文档摘要、章节摘要、关键概念和可复习问题。

对于音频文件，AI 可以先转录，再按时间段总结，并提取主题。

对于视频文件，AI 可以提取音轨转文字，再生成课程笔记、章节摘要和行动项。

对于图片，AI 可以进行 OCR，并生成图片内容描述。

AI 不应该只是“总结器”，而应该承担四类任务：

| AI 层级 | 说明 |
|---|---|
| 单篇 AI | 对一条笔记或一个文件进行摘要、分析、提取要点 |
| 整理 AI | 推荐标签、知识库、文件夹，发现重复和相似内容 |
| 知识库 AI | 基于某个知识库进行问答、总结和写作辅助 |
| 复习 AI | 根据历史笔记生成每日复习内容和主动回忆问题 |

产品要特别注意 AI 的可信度。知识库 AI 在回答问题时，应该引用来源笔记，避免用户不知道答案从哪里来。

---

### **6. 多端同步：手机端与网页端**

你明确提出手机和网页端同步，这也是产品必须认真设计的部分。

这里有两种路线。

第一种是**纯本地优先同步**，例如每个设备都有完整数据库，通过 CRDT 或同步协议合并冲突。这种方式很高级，也很符合开源精神，但实现复杂度高，尤其是移动端、文件资产、AI 处理结果、权限、多用户部署都会让难度上升。

第二种是**服务端同步 + 客户端本地缓存**。也就是用户的数据主要存储在服务器端，网页端和手机端通过 API 同步数据，同时手机端保留本地缓存，支持离线查看和简单编辑。这个方案更适合第一阶段落地。

我建议本项目早期采用第二种方案：

> 以服务端作为同步中心，网页端和手机端通过统一 API 同步；客户端保留本地缓存；核心内容支持 Markdown 导出；服务端可由用户自托管。

这样可以同时满足多端同步、开源自托管和数据可迁移。

---

## **六、推荐产品架构**

### **1. 端侧形态**

第一阶段建议优先做：

| 端 | 定位 | 优先级 |
|---|---|---|
| Web 网页端 | 主力使用端，负责编辑、整理、知识库、AI 问答 | P0 |
| Mobile 手机端 | 快速记录、查看、复习、上传图片/音频/文件 | P1 |
| 浏览器插件 | 网页剪藏、保存文章、保存链接 | P1 |
| 桌面端 | 可后续用 Web 技术封装 | P2 |

网页端是第一主战场。因为它最适合做复杂编辑、知识库管理、AI 面板和文件处理。

手机端第一版不需要做成完整的知识库管理工具，而应该聚焦四件事：

- 快速记录想法
- 上传图片、音频、文件
- 查看和搜索笔记
- 每日复习

也就是说，手机端不是一开始就复制完整网页端，而是作为移动捕捉和复习入口。

---

### **2. 服务端架构**

服务端应该负责账号、同步、数据存储、文件资产、AI 任务、搜索索引和权限控制。

建议拆成几个核心模块：

| 模块 | 作用 |
|---|---|
| API 服务 | 给网页端、手机端、插件提供统一接口 |
| 笔记服务 | 管理笔记的创建、编辑、删除、归档 |
| 标签服务 | 解析和管理 `#a/b/c` 分级标签 |
| 知识库服务 | 管理知识库、文件夹、权限 |
| 文件资产服务 | 管理视频、音频、图片、PDF 等原始文件 |
| AI 任务服务 | 处理摘要、转录、OCR、分类、向量化 |
| 搜索服务 | 全文搜索、标签搜索、语义搜索 |
| 同步服务 | 处理多端数据同步、版本、冲突 |
| 导入导出服务 | Markdown 导入导出、备份恢复 |

---

### **3. 存储架构**

为了兼顾开放性和产品能力，建议采用三层存储。

第一层是**Markdown 内容层**。用户的核心笔记正文应该可以导出为 Markdown。Markdown 是用户真正拥有的数据。

第二层是**元数据层**。标签、知识库、创建时间、来源、AI 摘要、复习状态、文件关联、向量索引等信息需要存储在数据库中。

第三层是**文件资产层**。视频、音频、图片、PDF 等原始文件不适合直接存在 Markdown 里，应该存储在对象存储或本地文件系统中，并在笔记中引用。

可以这样理解：

| 数据类型 | 存储方式 |
|---|---|
| 笔记正文 | Markdown |
| 笔记标题、时间、状态 | 数据库 |
| 标签、知识库、文件夹 | 数据库 |
| AI 摘要、关键词、分析 | 数据库或 Markdown frontmatter |
| 图片、音频、视频、PDF | 对象存储或本地文件系统 |
| 搜索索引 | 搜索引擎或数据库索引 |
| 向量数据 | 向量数据库或嵌入索引 |

一条笔记导出时，可以形成这样的结构：

```markdown
---
title: "AI 笔记系统产品想法"
created_at: "2026-05-09"
updated_at: "2026-05-09"
tags:
  - 产品/AI笔记/需求分析
  - 创业/开源项目
knowledge_base: "AI 产品研究"
source: "manual"
assets:
  - "./assets/audio-001.mp3"
  - "./assets/image-001.png"
---

这里是笔记正文内容。

#产品/AI笔记/需求分析 #创业/开源项目

这里是用户记录的想法。
```

这样设计的好处是，即使用户离开 OpenNote AI，也能用其他 Markdown 工具继续打开和管理自己的内容。

## **七、多端同步方案设计**

### **1. 同步目标**

多端同步要实现的是：

用户在手机端记录一条想法，网页端可以马上看到。用户在网页端整理标签和知识库，手机端可以同步更新。用户在手机端上传图片、音频或视频文件，服务端可以处理并生成 AI 笔记。用户离线时仍然能查看最近内容，并在恢复网络后同步修改。

第一阶段不必追求极端复杂的实时协同编辑，但要保证个人多端同步稳定可靠。

---

### **2. 推荐同步模型**

建议采用：

> 服务端权威数据 + 客户端本地缓存 + 增量同步 + 版本号冲突处理。

具体来说，服务端保存最新版本的数据。每个客户端保存一份本地缓存。客户端启动时拉取服务端更新，用户编辑后提交变更到服务端。服务端根据版本号判断是否冲突。

每条笔记可以有这些同步字段：

| 字段 | 说明 |
|---|---|
| id | 全局唯一 ID |
| user_id | 用户 ID |
| title | 标题 |
| content | Markdown 正文 |
| version | 版本号 |
| updated_at | 更新时间 |
| deleted_at | 软删除时间 |
| sync_status | 本地同步状态 |
| device_id | 最后修改设备 |
| content_hash | 内容校验值 |

移动端本地可以使用 SQLite 或类似本地数据库缓存笔记。网页端可以使用 IndexedDB 做本地缓存。

---

### **3. 冲突处理策略**

第一阶段可以采用简单、可理解的冲突策略。

如果用户在手机端和网页端同时编辑同一条笔记，系统不要直接覆盖，而是创建冲突副本。

例如：

- 原笔记：《AI 笔记系统想法》
- 冲突副本：《AI 笔记系统想法 - 来自手机端的冲突版本》

用户可以手动合并。

这比复杂自动合并更安全，也更容易实现。

后续如果要支持多人协作或实时编辑，再考虑 CRDT 或 OT 协同算法。

对于第一阶段个人笔记系统来说，冲突副本已经足够。

---

### **4. 文件资产同步**

文件资产比普通笔记更复杂，尤其是视频和音频文件。

建议采用以下策略：

笔记正文和元数据优先同步，文件资产异步上传和处理。

例如用户在手机端上传一个视频文件后，系统先创建一条资产记录，状态为“上传中”。上传完成后，服务端开始转码、提取音轨、转写和 AI 摘要。网页端可以看到该资产的处理状态。

文件资产可以有这些状态：

| 状态 | 说明 |
|---|---|
| pending | 等待上传或等待处理 |
| uploading | 上传中 |
| uploaded | 上传完成 |
| processing | AI 处理中 |
| completed | 处理完成 |
| failed | 处理失败 |

对于大文件，建议支持断点续传和后台上传。手机端尤其需要这个能力，否则上传视频和长音频体验会很差。

第一版可以先限制文件大小，例如先支持 100MB 以内文件，后续再放开。

---

### **5. 手机端同步体验**

手机端的产品重点是快。

用户打开手机端，应该能直接看到一个输入框：

“记录你的想法……”

用户输入后立即保存到本地，同时后台同步到服务端。如果网络不好，状态显示为“待同步”，网络恢复后自动上传。

手机端还应该支持系统分享入口。用户在其他 App 看到网页、图片、PDF、音频或视频时，可以通过系统分享菜单发送到 OpenNote AI。

手机端首页建议包括：

- 快速记录框
- 最近笔记
- 待同步内容
- 今日复习
- 上传入口
- 搜索入口

不要在第一版手机端做太复杂的知识库管理，否则会拖慢开发进度。

---

### **6. 网页端同步体验**

网页端是主力整理端。

用户在网页端可以查看 Inbox、编辑 Markdown、管理知识库、调整标签、查看 AI 摘要、处理文件资产、进行知识库问答。

网页端应该支持自动保存。用户编辑时，系统定期保存草稿，并显示同步状态：

- 已保存
- 正在保存
- 同步失败
- 离线编辑中
- 存在冲突

网页端还可以作为文件资产的主要处理入口。比如用户上传 PDF、音频、视频后，在网页端查看转写结果和 AI 总结。

---

## **八、核心功能模块**

### **1. Inbox 收集箱**

Inbox 是所有内容的入口。

无论是手动输入的想法，还是网页剪藏、PDF、图片、音频、视频文件，默认都先进入 Inbox。

Inbox 的作用不是长期归档，而是暂存、处理和整理。

每条 Inbox 内容应该展示：

| 信息 | 说明 |
|---|---|
| 标题 | 用户输入或 AI 自动生成 |
| 类型 | 文字、网页、PDF、图片、音频、视频 |
| 来源 | 手动输入、上传文件、分享、网页剪藏 |
| 摘要 | AI 自动生成 |
| 推荐标签 | AI 推荐的 `#a/b/c` 标签 |
| 推荐知识库 | AI 推荐归属 |
| 处理状态 | 待处理、处理中、已完成、失败 |
| 整理状态 | 未整理、已归档 |

Inbox 的核心交互是：

用户把内容丢进来，AI 帮用户初步处理，用户最后确认分类。

---

### **2. 快速记录**

快速记录是产品的高频入口。

它应该像 flomo 一样简单：打开就写，写完就保存。

用户可以直接输入：

```text
今天想到一个 AI 笔记系统的核心差异点，不只是收藏信息，而是每天帮我复习旧内容，让知识重新被激活。#产品/AI笔记/复习系统
```

系统保存后自动完成：

- 解析标签 `#产品/AI笔记/复习系统`
- 自动生成标题
- 自动生成摘要
- 推荐知识库
- 进入 Inbox 或直接归档

如果用户没有写标签，AI 可以建议标签，但不强制。

---

### **3. Markdown 编辑器**

Markdown 是产品的核心内容格式。

编辑器不需要一开始做得像 Notion 那样复杂，但要足够好用。

第一版应支持：

- Markdown 编辑
- Markdown 预览
- 自动保存
- 快捷键
- 图片粘贴
- 文件引用
- 标签高亮
- 代码块
- 标题目录
- 双链或笔记链接，后续可做

编辑器里出现的 `#a/b/c` 标签应该自动高亮，并可点击跳转到标签页。

---

### **4. flomo 式分级标签系统**

标签系统是本产品的重要特色之一。

用户在任何笔记正文中输入 `#` 开头的内容，系统自动识别为标签。

例如：

```text
这篇文章讲的是 AI 笔记如何从收藏工具变成复习系统。#产品/AI笔记/定位 #知识管理/复习
```

系统会自动解析出两个标签：

- `#产品/AI笔记/定位`
- `#知识管理/复习`

同时，系统也应该支持父级聚合。

点击 `#产品` 可以看到所有：

- `#产品/AI笔记`
- `#产品/需求分析`
- `#产品/竞品分析`
- `#产品/用户研究`

点击 `#产品/AI笔记` 可以看到其下所有更细分的标签和笔记。

标签页面可以设计为左右结构。

左侧是标签树，右侧是对应笔记流。

标签树示例：

```text
#产品
  /AI笔记
    /定位
    /需求分析
    /竞品分析
  /用户研究
  /商业化

#知识管理
  /复习
  /输入
  /输出
```

但要注意，标签树是系统根据用户笔记自动生成的，不要求用户提前维护。

---

### **5. 知识库系统**

知识库是长期主题空间。

一个知识库可以包含文件夹、笔记、文件资产和知识库 AI。

例如“AI 产品研究”知识库中可以有：

```text
AI 产品研究
├── 竞品分析
├── 用户需求
├── 产品设计
├── 技术方案
├── 商业化
└── 灵感记录
```

知识库适合承载更稳定、更系统的内容。

用户可以手动创建知识库，也可以让 AI 根据长期标签和笔记内容建议创建知识库。

知识库页面应该包含：

- 知识库目录
- 最近更新笔记
- 知识库内搜索
- 知识库 AI 问答
- 相关标签
- 文件资产
- 复习进度

---

### **6. 文件资产管理**

这是你补充后非常重要的一块。

OpenNote AI 需要把文件资产作为一等公民，而不是附件而已。

文件资产包括：

- 图片
- 截图
- PDF
- 音频
- 视频
- 文档
- 压缩包，后续可选
- 其他资料文件

每个文件资产都应该有独立记录，并能关联到一条或多条笔记。

例如用户上传一个课程视频，系统可以生成：

- 原始视频文件
- 自动转写稿
- AI 课程摘要
- 章节大纲
- 关键概念
- 可复习问题
- 关联标签
- 所属知识库

这条内容最终可以表现为一条 Markdown 笔记，但原视频仍然作为资产保留。

文件资产的数据结构可以包括：

| 字段 | 说明 |
|---|---|
| asset_id | 文件资产 ID |
| file_name | 文件名 |
| file_type | 文件类型 |
| mime_type | MIME 类型 |
| file_size | 文件大小 |
| storage_path | 存储路径 |
| source | 上传来源 |
| linked_note_id | 关联笔记 |
| processing_status | 处理状态 |
| transcript | 音视频转录文本 |
| ocr_text | 图片 OCR 文本 |
| ai_summary | AI 摘要 |
| created_at | 创建时间 |

这样做的好处是，系统既能保留原始文件，又能把文件内容转化成可搜索、可总结、可问答的知识内容。

---

### **7. AI 单篇总结与分析**

每条笔记或文件都可以生成 AI 分析。

建议第一版支持以下字段：

| AI 结果 | 说明 |
|---|---|
| 自动标题 | 为无标题内容生成标题 |
| 摘要 | 简要概括内容 |
| 核心要点 | 提取 3 到 7 条重点 |
| 关键词 | 提取主题词 |
| 推荐标签 | 推荐 `#a/b/c` 格式标签 |
| 推荐知识库 | 建议归属知识库 |
| 行动项 | 提取待办或下一步 |
| 复习问题 | 生成用于主动回忆的问题 |

对于视频和音频文件，AI 流程是：

> 文件上传 → 提取音频 → 语音转文字 → 分段 → 摘要 → 标签建议 → 复习问题 → 生成 Markdown 笔记。

对于图片和截图，AI 流程是：

> 图片上传 → OCR → 图片描述 → 关键信息提取 → 标签建议 → 生成 Markdown 笔记。

---

### **8. AI 自动整理**

AI 自动整理的目标是减少用户整理负担，但不能让用户失控。

建议第一阶段采用“建议确认制”。

AI 可以推荐：

- 这条笔记应该放进哪个知识库
- 应该打哪些分级标签
- 是否与已有笔记重复
- 是否与某些旧笔记相关
- 是否应该加入每日复习
- 是否应该归档

用户可以一键确认，也可以修改。

例如：

```text
AI 建议：

推荐知识库：AI 产品研究

推荐标签：
#产品/AI笔记/功能设计
#知识管理/复习系统
#开源项目/产品规划

推荐理由：
这条笔记主要讨论 AI 笔记系统的功能结构、复习机制和开源产品方向。
```

等用户长期使用后，可以增加自动规则。

例如：

- 包含 `#产品/AI笔记` 的笔记自动进入“AI 产品研究”知识库
- 上传的 PDF 默认进入“待读资料”
- 所有语音输入默认进入 Inbox
- 每天晚上自动整理当天新笔记

---

### **9. 知识库 AI**

知识库 AI 是产品从“笔记工具”升级为“个人知识助理”的关键。

用户可以对某个知识库提问：

```text
我之前关于 AI 笔记系统的核心想法有哪些？
```

```text
帮我整理一个“开源 AI 笔记系统”的产品框架。
```

```text
我在哪些笔记里提到过复习系统？
```

```text
根据我的笔记，帮我生成一份 README。
```

知识库 AI 的回答必须尽量引用来源笔记。

例如：

```text
根据你的笔记，你对 AI 笔记系统的核心想法主要有三点：

第一，它应该把外部信息和个人想法统一沉淀为 Markdown。
来源：《AI 笔记系统初始想法》

第二，它应该使用 flomo 式分级标签，而不是复杂数据库字段。
来源：《标签系统设计》

第三，它应该通过每日复习激活旧知识。
来源：《复习系统产品设想》
```

引用来源可以显著提高用户信任感。

---

### **10. 每日复习系统**

复习系统是本项目非常重要的差异化能力。

很多笔记软件只是把内容存起来，但用户很少再打开。OpenNote AI 应该主动帮助用户重新遇见过去的内容。

每日复习可以有几种模式。

第一种是时间回顾。系统每天推荐几条过去的笔记，例如 3 天前、7 天前、30 天前、180 天前的内容。

第二种是 AI 提问。系统根据笔记生成问题，让用户主动回忆，而不是被动阅读。

第三种是主题复习。用户可以选择复习某个知识库或某个标签下的内容。

第四种是记忆卡片。系统从笔记中生成问答卡片，用户标记掌握程度。

复习状态可以包括：

| 状态 | 说明 |
|---|---|
| 未复习 | 还没有进入复习流程 |
| 今日待复习 | 今天需要回顾 |
| 已掌握 | 用户认为已经熟悉 |
| 模糊 | 用户部分记得 |
| 忘记 | 用户基本忘记 |
| 已暂停 | 不再主动复习 |

第一版不需要做复杂算法，可以先做简单规则：

- 新笔记创建后第 1 天复习
- 第 3 天复习
- 第 7 天复习
- 第 14 天复习
- 第 30 天复习

后续再根据用户反馈调整复习间隔。

---

## **九、核心页面设计**

### **1. 首页 / 今日页**

首页是用户每天进入产品后看到的地方。

它应该包含：

- 快速记录框
- 今日待复习
- 最近笔记
- 待整理 Inbox
- AI 整理建议
- 最近上传文件
- 搜索入口

首页的目标不是展示所有功能，而是帮助用户马上开始记录、复习或整理。

---

### **2. Inbox 页面**

Inbox 是收集箱。

这里展示所有未整理内容，包括文字、网页、PDF、图片、音频、视频。

每条内容以卡片形式展示：

- 标题
- 类型
- 来源
- AI 摘要
- 推荐标签
- 推荐知识库
- 处理状态
- 整理按钮

用户可以批量确认 AI 建议，把内容归档到知识库。

---

### **3. 笔记详情页**

笔记详情页建议采用三栏结构。

左侧是知识库或目录，中间是 Markdown 编辑器，右侧是 AI 面板。

右侧 AI 面板包括：

- 摘要
- 核心要点
- 推荐标签
- 推荐知识库
- 相关笔记
- 复习问题
- 文件资产
- AI 问答入口

如果是视频或音频笔记，还可以显示转录稿和时间段摘要。

---

### **4. 标签页**

标签页采用 flomo 式标签体验，但可以增强浏览能力。

页面包括：

- 标签搜索
- 标签树
- 标签下笔记流
- 子标签列表
- 标签重命名
- 标签合并

点击 `#产品/AI笔记` 后，用户可以看到所有相关笔记，并可以继续筛选子标签。

---

### **5. 知识库页**

知识库页是系统化沉淀的地方。

包括：

- 知识库目录
- 文件夹
- 笔记列表
- 文件资产列表
- 知识库 AI
- 相关标签
- 最近更新
- 复习进度

用户可以在这里对某个主题进行深度学习和整理。

---

### **6. 文件资产页**

文件资产页用于集中管理上传的文件。

用户可以按类型筛选：

- 图片
- PDF
- 音频
- 视频
- 文档

每个文件资产显示：

- 文件名
- 类型
- 大小
- 上传时间
- 处理状态
- 关联笔记
- AI 摘要

这个页面对重度资料用户很重要，尤其是课程、录音、PDF 和截图较多的用户。

---

### **7. 复习页**

复习页是产品差异化页面。

每日复习流程可以是：

用户进入复习页，系统展示一条旧笔记生成的问题。

例如：

```text
问题：你为什么认为 AI 笔记系统不应该只做收藏，而应该做复习？
```

用户先尝试回答，然后点击查看原笔记和 AI 总结。最后用户标记：

- 记得
- 有点模糊
- 忘了
- 不再复习

系统根据反馈安排下次复习。

---

## **十、MVP 版本规划**

### **v0.1：AI Markdown Inbox**

第一版目标是做出最小可用闭环：

> 快速记录 → Markdown 保存 → flomo 式标签解析 → AI 摘要 → Inbox 整理 → 知识库归档 → 多端基础同步。

v0.1 不追求支持所有文件类型，不追求复杂 AI，不追求移动端全功能。

v0.1 必须完成：

| 模块 | 功能 |
|---|---|
| 账号系统 | 登录、个人空间 |
| Web 端 | 基础笔记管理 |
| 快速记录 | 打开即写 |
| Markdown 编辑 | 编辑、预览、自动保存 |
| 分级标签 | 支持 `#a/b/c` 解析、点击、筛选 |
| Inbox | 新内容默认进入收集箱 |
| 知识库 | 创建知识库、归档笔记 |
| AI 摘要 | 自动标题、摘要、关键词、标签建议 |
| 搜索 | 全文搜索、标签搜索 |
| 同步 | Web 与服务端同步 |
| 导出 | Markdown 导出 |

v0.1 的核心验证点是：

用户是否愿意把日常想法写进来，是否觉得标签自然，是否觉得 AI 摘要和分类建议有价值。

---

### **v0.2：文件资产基础版**

v0.2 开始支持文件资产。

建议支持：

| 类型 | 功能 |
|---|---|
| 图片 | 上传、OCR、生成笔记 |
| PDF | 上传、文本提取、AI 摘要 |
| 文档 | TXT、MD、DOCX 导入 |
| 附件 | 文件与笔记关联 |
| 文件资产页 | 按类型管理文件 |

这一版的目标是让 OpenNote AI 从“笔记工具”变成“资料入口”。

---

### **v0.3：移动端捕捉与复习**

v0.3 开始做手机端。

手机端优先支持：

- 快速记录
- 查看最近笔记
- 搜索笔记
- 上传图片和文件
- 每日复习
- 离线缓存
- 后台同步

这版不必实现完整知识库管理。

目标是让用户在手机上也能完成高频动作。

---

### **v0.4：音频与视频文件处理**

这一版重点处理音视频资产。

支持：

- 音频上传
- 视频上传
- 音轨提取
- 语音转文字
- 分段摘要
- 章节大纲
- AI 复习问题

## **十二、开源与自托管设计**

OpenNote AI 应该从第一天就考虑开源和自托管。

建议产品承诺：

- 用户可以自托管服务端
- 用户可以导出 Markdown
- 用户可以备份数据库
- 用户可以选择自己的 AI 服务
- 用户可以关闭 AI 功能，只使用基础笔记
- 用户可以迁移文件资产
- 项目提供 Docker 部署方式

AI 服务应该可插拔。

用户可以选择：

- 云端大模型 API
- 自己配置的模型服务
- 本地模型，后续支持

这样可以避免用户再次被某一个 AI 服务绑定。

## **十三、产品体验原则**

### **1. 先记录，后整理**

用户记录时不要被分类、标题、文件夹打断。所有内容都可以先进入 Inbox，后续再整理。

### **2. 标签自然生长**

标签不是数据库字段优先，而是正文内容优先。用户写下 `#a/b/c`，系统自动理解。

### **3. Markdown 是核心资产**

Markdown 不是导出格式，而是核心内容格式。用户的知识应该长期可迁移。

### **4. AI 给建议，用户做决定**

AI 可以总结、分类、推荐标签、推荐知识库，但不要默默替用户做不可逆操作。

### **5. 文件资产与笔记融合**

视频、音频、PDF、图片不是普通附件，而是可以被解析、总结、搜索、问答和复习的知识资产。

### **6. 手机端负责捕捉，网页端负责整理**

第一阶段不要让手机端承担过多复杂功能。移动端要快，网页端要强。

### **7. 复习是核心闭环**

笔记不只是保存，而是要被重新激活。每日复习应该成为产品的重要入口。

## **十四、竞品差异化方向**

OpenNote AI 的差异化不应该只是“开源版某某产品”，而应该形成自己的定位。

与 flomo 相比，OpenNote AI 更强调文件资产、AI 总结、知识库和自托管。

与 Notion 相比，OpenNote AI 更轻量、更 Markdown、更适合快速记录和 AI 自动整理。

与 Obsidian 相比，OpenNote AI 更强调云端同步、手机捕捉、AI 原生和文件自动处理。

与普通 AI 知识库相比，OpenNote AI 更适合日常笔记、碎片想法、标签体系和长期复习。

与商业 AI 笔记产品相比，OpenNote AI 的优势是开源、可自托管、可导出、可扩展。

## **十六、风险与挑战**

### **1. 范围过大**

你现在的产品愿景很完整，但功能面非常广。如果一开始同时做网页端、手机端、视频、音频、PDF、AI、同步、复习，很容易失控。

解决方式是严格按版本推进。第一版只做 AI Markdown Inbox。

### **2. 音视频处理成本高**

视频和音频转写需要消耗大量算力和存储。开源自托管场景下，用户的机器配置也不一定足够。

解决方式是把音视频处理放到 v0.4，并允许用户自己配置 AI 服务和转写服务。

### **3. 多端同步复杂**

同步看起来只是“手机和网页数据一样”，但实际涉及离线、冲突、大文件、权限、版本和失败重试。

解决方式是第一阶段采用服务端同步中心，不一开始做复杂实时协同。

### **4. AI 结果不稳定**

AI 可能总结错误、分类不准、标签混乱。

解决方式是所有 AI 动作先采用建议制，用户确认后再生效。

### **5. 标签体系可能膨胀**

`#a/b/c` 很自由，但长期使用后可能出现重复标签，例如 `#AI笔记`、`#AI/笔记`、`#产品/AI笔记`。

解决方式是提供标签合并、重命名和 AI 标签清理建议。

第一，Web 端快速记录。

第二，Markdown 编辑和保存。

第三，`#a/b/c` 分级标签解析。

第四，Inbox 收集箱。

第五，知识库基础归档。

## **十八、v0.1 用户流程**

### **流程一：记录想法**

用户打开网页端。

看到快速记录框。

输入：

```text
我想做一个开源 AI 笔记系统，核心不是收藏，而是把所有信息转成 Markdown，并每天复习。#产品/AI笔记/定位
```

点击保存。

系统自动：

- 保存为 Markdown
- 解析标签
- 生成标题
- 生成摘要
- 放入 Inbox
- 推荐知识库“AI 产品研究”

用户点击确认归档。

---

### **流程二：整理 Inbox**

用户进入 Inbox。

看到一批未整理笔记。

每条笔记都有 AI 建议。

用户批量确认：

- 添加推荐标签
- 归入推荐知识库
- 加入复习计划

整理完成后，笔记从 Inbox 移动到知识库。

---

### **流程三：每日复习**

第二天用户打开首页。

看到“今日复习”。

系统展示昨天的笔记，并生成问题：

```text
为什么这个产品的核心不是收藏，而是复习？
```

用户回答后查看原笔记。

系统根据用户反馈安排下次复习。

---

### **流程四：手机端后续体验**

用户在手机上突然有灵感。

打开 App，直接写一条内容。

保存后本地立即记录，并后台同步。

回到电脑网页端后，这条笔记出现在 Inbox 中，AI 已经生成摘要和标签建议。

## **十九、修改说明：这版相比上一版做了什么变化**

这版报告主要做了三处关键调整。

第一，我把“信息”扩展成了“外部信息与文件资产”。这样产品不再只处理网页、文章、链接，也能处理视频文件、音频文件、PDF、图片、截图等长期资料。这更符合你真实的知识管理需求，也让产品未来有更大的使用场景。

第二，我明确把标签系统设计成 flomo 式 `#a/b/c` 分级标签。这样可以避免做成传统标签系统，也能保留你喜欢的轻量记录体验。这个标签系统会自然存在于正文中，由系统自动解析、聚合和管理。

第三，我补充了手机端与网页端的多端同步方案。建议早期采用“服务端同步 + 本地缓存 + 增量同步 + 冲突副本”的方式。这样比纯本地优先同步更容易落地，也能支持开源自托管和后续移动端扩展。

## **二十、最终产品定义**

OpenNote AI 是一个开源 AI 笔记与个人知识管理系统。

它支持用户快速记录想法，收集网页、文章、PDF、图片、音频、视频等外部信息和文件资产，并将其统一转化为 Markdown 笔记。系统使用 flomo 式 `#a/b/c` 分级标签和知识库体系帮助用户组织内容，通过 AI 自动生成摘要、分析、标签建议、分类建议和复习问题。用户可以在网页端进行深度整理，在手机端快速记录和复习，并通过服务端实现多端同步。所有核心内容支持 Markdown 导出，系统支持开源自托管，避免用户被商业平台锁定。

它的核心价值可以浓缩成一句话：

> OpenNote AI 让你的想法、资料和文件资产变成真正属于你的、可整理、可问答、可复习、可迁移的个人知识系统。
